## 第II章 AI利用の指針(続き・5ページ目)

**● フェーズの到達目標:**

- 汎用的なプロンプト技術と、特定分野向けのプロンプト技術の両方を身につける。
- AIツールを効果的に使って生産性を高める。
- 計算論的思考、問題解決力、適応力、協働力を引き続き強化する。

**● コメントと例:**

- 試験や評価など、本当の理解を示さなければならない場面に今後も繰り返し遭遇します。技術スキルと対人スキルの両方を磨き続け、備えておきましょう。
- 自分の考えを説明し、仲間と議論することで、理解の穴が見つかることがよくあります。ピアラーニングを優先しましょう。
- AIツールはあなた固有の状況を把握しておらず、一般的な回答をしがちです。同じ環境を共有する仲間の方が、より的確で適切な知見を提供できます。
- AIが「最も確からしい答え」を生成するのに対し、仲間は別の視点や貴重なニュアンスを提供してくれます。品質チェックの場として仲間を頼りましょう。

✓ **良い実践例:**
AIに「ソート関数はどうテストすればいい?」と質問する。いくつかのアイデアをもらい、試してみて、その結果を仲間とレビューする。一緒にアプローチを磨き上げる。

✗ **悪い実践例:**
AIに関数全体を書かせ、プロジェクトにコピペする。ピア評価のとき、それが何をするのか、なぜそうなのか説明できない。信頼を失い、プロジェクトも不合格になる。

✓ **良い実践例:**
AIを使ってパーサーの設計を手伝ってもらう。その後、仲間とロジックを一緒に確認する。バグを2つ見つけ、一緒に書き直す——より良く、よりきれいに、完全に理解した上で。

✗ **悪い実践例:**
プロジェクトの重要な部分のコードをCopilotに生成させる。コンパイルは通るが、パイプをどう処理しているのか説明できない。評価で正当な説明ができず、プロジェクトに落ちる。

## 第IV章 演習0: データプロセッサ(7〜9ページ目)

| | Exercise0 |
|---|---|
| | data_processor |
| ディレクトリ | ex0/ |
| 提出ファイル | data_processor.py |
| 許可 | builtins、標準型、import typing、import abc |

**エンジニアリング・ブリーフィング**: Code Nexus へようこそ!データ処理システムの基盤を構築せよ。異なるデータ型が、それぞれ固有の特性を保ちつつ、共通の処理インターフェースを共有できることを示す、ベースとなるプロセッサ・アーキテクチャを作成する。

この演習では `ABC`(抽象基底クラス)を用いた*抽象クラス*の使用が必須である。まずは共通インターフェースを持つ個別のクラスを作成する。次の演習で、それらを同一のワークフローに統合する。

以下のアーキテクチャを構築すること:

- `ABC` を継承し、共通の処理インターフェースを定義する抽象クラス `DataProcessor`。
- `DataProcessor` クラスを継承し、それぞれ異なる種類のデータを処理する3つの特化クラス `NumericProcessor`、`TextProcessor`、`LogProcessor`。
- `DataProcessor` 内の2つの抽象メソッド: 入力データが現在のデータプロセッサに適しているかを確認する `validate` と、入力データを処理する `ingest`。各特化クラスはこれらのメソッドをオーバーライドする必要がある。
- `DataProcessor` 内の1つの通常メソッド: 取り込み済みデータを出力する `output`。

以下の制約に従うこと:

- `validate` メソッドは `DataProcessor` クラスで `validate(self, data: Any) -> bool` として定義する。特化クラス側のオーバーライドメソッドも同じシグネチャを持つ。どんなデータが送られてくるか分からないため、任意の型を受け取れなければならないからである。このメソッドは、与えられたデータをこのデータプロセッサが取り込めるかどうかを示す bool を返す。
- `ingest` メソッドは `DataProcessor` クラスで `ingest(self, data: Any) -> None` として定義する。特化クラス側のオーバーライドメソッドは、期待する型に合わせた独自のシグネチャを持つ。ユーザーが `validate` せずに `ingest` を呼び、無効なデータを渡した場合は、例外を送出しなければならない。
- `output` メソッドは `DataProcessor` クラスで `output(self) -> tuple[int, str]` として定義する。特化クラスでオーバーライドする必要はない。
- `NumericProcessor` は `int`、`float`、および両者のリスト(混在リストを含む)を取り込む。データを文字列に変換して内部に保存し、`output` メソッドで取り出されるのを待つ。オーバーライドする `ingest` のシグネチャは受け付ける型を反映すること。
- `TextProcessor` は `str` と文字列のリストを取り込む。データを内部に保存し、`output` メソッドで取り出されるのを待つ。オーバーライドする `ingest` のシグネチャは受け付ける型を反映すること。
- `LogProcessor` は文字列のキーと値からなる `dict`、およびそのリストを取り込む。データを文字列に変換して内部に保存し、`output` メソッドで取り出されるのを待つ。オーバーライドする `ingest` のシグネチャは受け付ける型を反映すること。
- `output` メソッドは、データプロセッサ内部に保存された最も古いデータを、そのプロセッサ内での処理順位(rank)とともに取り出す。取り出されたデータはプロセッサから削除される。

最後に、アーキテクチャをテストすること:

- 各特化クラスのインスタンスを作成する。
- `validate` メソッドを通して、各クラスで有効なデータと無効なデータをテストする。
- 事前の検証なしに、少なくとも1つの無効なデータを `ingest` メソッドに渡し、例外が送出されることを確認する。これにより `mypy` の警告が残るが、それは意図的なものである。
- 各データプロセッサに様々なデータを取り込み、`output` で取り出す。

(9ページ目の実行例は英語のまま動作出力なので省略可ですが、内容は「数値・テキスト・ログ各プロセッサの validate/ingest/output のテスト出力例」です。)

**エンジニアリング・ブリーフィング**: 複数のデータ型を同時に扱える、適応型のストリーム処理ワークフローを構築する時が来た。

演習0のコードを使い、以下のように改良すること:

- 異なる型を含むデータストリームを受け取り、ポリモーフィックな振る舞いを利用して各要素を適切なデータプロセッサへ振り分ける `DataStream` クラスを作成する。
- このクラスは、データストリームを処理する新しいデータプロセッサを登録できる
  `def register_processor(self, proc: DataProcessor) -> None:`
  メソッドを実装する。
- このクラスは、引数として受け取ったリストの各要素を解析し、適切な登録済みデータプロセッサへ送る
  `def process_stream(self, stream: list[typing.Any]) -> None:`
  メソッドを実装する。どのデータプロセッサも処理できない要素があれば、エラーメッセージを表示する。
- 最後に、ストリームの統計を表示する `def print_processors_stats(self) -> None:` メソッドを実装する。
- データストリームが正しく処理されることを示すテストシナリオを作成すること。登録済みデータプロセッサの統計を表示し、各データプロセッサの `output` メソッドで要素を消費し、更新後の統計を表示する。

## 第VI章 演習2: データパイプライン(12〜13ページ目)

| | Exercise2 |
|---|---|
| | data_pipeline |
| ディレクトリ | ex2/ |
| 提出ファイル | data_pipeline.py |
| 許可 | builtins、標準型、import typing、import abc |

**エンジニアリング・ブリーフィング**: 最終課題は、これまでのすべてを統合し、エンタープライズ規模でのポリモーフィック・アーキテクチャの習熟を示す、完全なデータ処理パイプラインを構築することである。

演習1のコードを改良し、完全なデータパイプラインを完成させること。`DataStream` クラスはすでに入力ストリームを正しく処理できている。次はパイプラインの出力側を扱う。これは、*ダックタイピング*によってエクスポート互換となる、エクスポートクラスのプラグインシステムを使って実現する。

以下を実装すること:

- 特殊な `Protocol` クラスを継承する新しい `ExportPlugin` クラス。
- このクラスは、各エクスポートプラグインに対する制約として機能する以下のメソッドを定義する:
  `def process_output(self, data: list[tuple[int, str]]) -> None:`
  `data` パラメータの型は、`DataProcessor` クラスの `output` メソッドの戻り値に対応するタプルのリストである。
- `DataStream` クラスは新たに
  `def output_pipeline(self, nb: int, plugin: ExportPlugin) -> None:`
  メソッドを実装する。これは `process_stream` の呼び出し後に使用し、登録済みのすべてのデータプロセッサから `nb` 個の要素を消費して、与えられた互換プラグインでエクスポートする。
- 少なくとも CSV エクスポートプラグインと JSON エクスポートプラグインを作成すること。これらのプラグインのために特別なインポートを使う必要はない。有効な CSV / JSON 文字列を手動で組み立てること。
